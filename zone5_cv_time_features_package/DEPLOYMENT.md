## Runtime Pipeline

Keep these collectors running while building live data:

```text
RTSP camera CV person counter
    -> ./run_person_counter.ps1 or ./run_person_counter.sh
    -> cv_counter package-local script, mask, model, and ByteTrack config
    -> data/yolo_latest.jpg masked annotated frame for the web app
    -> MQTT topic care_ssl/zone5/person_count

CV person-count MQTT
    -> python -m zone5.occupancy_mqtt_aggregator
    -> data/cv_occupancy_zone5_10sec.csv

SEN55 MQTT
    -> python -m zone5.sen55_mqtt_collector
    -> data/sen55_data.csv (one row per completed 10-second bucket)

AIR-1 API + smart plug API + mmWave API
    + data/sen55_data.csv
    + data/cv_occupancy_zone5_10sec.csv
    -> python -m zone5.collect_training_data --live-append
    -> data/zone5_training_cv.csv
    -> data/zone5_training_cv.metadata.json
```

In the systemd deployment, training is a separate scheduled job:

```text
zone5-trainer.timer
    -> zone5-trainer.service
    -> bash run_zone5_trainer.sh
    -> stable snapshot in data/training_snapshots/
    -> model/runs/<run_id>/ and model/current_run.txt

model/current_run.txt
    -> automatic python -m zone5.promote_model after each successful timer retrain
    -> model/production_run.txt
```

Fresh packages do not ship `model/current_run.txt` or
`model/production_run.txt`. Training creates the current pointer; promotion
creates the production pointer used by the app.

Persistent deployment can start from a fresh folder. Start the collectors and
the trainer timer immediately; the collectors create the data files first. The
trainer may log hourly skips or failures until the joined table has enough
blind-test positive evidence, enough coverage-eligible calendar days, and
usable train/validation windows. Those early failures do not stop data
collection. After a retrain succeeds, automatic promotion runs and updates
`model/production_run.txt` only if the candidate passes the promotion checks.

The app can also run from the start:

```text
model/production_run.txt
    + live AIR-1/API data
    + data/sen55_data.csv
    -> web_app
    -> http://127.0.0.1:8000/
```

Before the first successful promotion, the app reports `production pointer
missing`. After automatic promotion writes the pointer, the app reloads it on
its pointer poll interval.

## Runtime CSVs And Training Snapshots

Runtime services are CSV-first. The web app reads live API data plus
`data/sen55_data.csv` and `data/cv_occupancy_zone5_10sec.csv`; training wrappers
read `data/zone5_training_cv.csv`.

`data/cv_occupancy_zone5_10sec.csv` is updated as each completed 10-second
bucket passes the default 5-second late grace period.

`data/sen55_data.csv` is written from an in-memory 10-second bucket buffer.
Numeric SEN55 fields are averaged in each bucket, metadata fields use the
latest non-empty value, and payloads for already-flushed buckets are skipped.

`data/zone5_training_cv.csv` is appended every live collector cycle. Each live
append cycle refetches the latest 120 seconds by default so late SEN55 samples
and CV occupancy labels can backfill existing rows.
`data/zone5_training_cv.metadata.json` is refreshed on the configured interval.
Direct collector runs do not retrain unless `--retrain-after-snapshot` is
passed. For the systemd deployment, keep `RETRAIN_AFTER_SNAPSHOT=0` on
`zone5-live-collector.service` and run
`zone5-trainer.service`/`zone5-trainer.timer` separately so training and
promotion cannot block live appends.

The separate trainer uses `run_zone5_trainer.sh`, snapshots the live CSV by
default into immutable Parquet files under `data/training_snapshots/`, takes
`model/retrain.lock` to prevent overlapping runs, trains with the same
bootstrap/progressive strict-CV policy, and promotes only when the candidate
passes the configured gates. No runtime service writes a top-level
`data/*.parquet` cache. The default bootstrap setting enables fallback only
while `model/production_run.txt` is missing or empty. The default strict-CV
setting progresses validation from 1 to 2 to 3 one-day folds as production
matures. Hourly retraining can skip or fail during the first collection period
because training needs enough blind-test positive evidence, at least two
coverage-eligible pre-test dates for strict rolling validation, and usable
occupied and unoccupied labels in the relevant windows. The collector keeps
appending data while trainer logs go to `logs/zone5_trainer.log`.

### 1. Install And Prepare


Dependencies are installed into `.python-packages/` inside this folder. That
keeps this package separate from user/global Python packages and avoids
conflicts with tools such as `emuvim`. The person counter uses Ultralytics
ByteTrack by default and can be switched to BoT-SORT, so `requirements.txt`
includes `lap>=0.5.12`; installing from `requirements.txt` here prevents
Ultralytics from trying to auto-install `lap` into `/usr/local` when the
service starts. The person-counter launcher
also sets `YOLO_AUTOINSTALL=False` and checks `import lap` before opening the
camera, so a missing dependency fails with a clear install command.

Enable user-service linger once so `systemctl --user` services keep running
after you disconnect from SSH and can restart after reboot:

```bash
loginctl enable-linger "$USER"
```

If Ubuntu denies that command, run it with sudo:

```bash
sudo loginctl enable-linger "$USER"
```

### 5. Check Live Inference Output

Treat service reachability and inference output as separate checks.
`/api/health` with `ok=true`, the dashboard page, and video availability prove
that the live app is reachable. `/api/current.probability` proves the live app
is producing an inference probability. Check `/api/current` after `/api/health`
and inspect `error` if `probability` is missing.

```bash
curl http://127.0.0.1:8000/api/current
```

If `/api/current` reports:

```text
LIVE DATA DEGRADED: core sensor coverage below gate
```

then live prediction is blocked by core sensor coverage. The gate requires at
least `0.80` coverage for AIR-1 and power fields, and at least `0.95` coverage
for `mmwave_s5`. `sen55-missing` is not the blocker by itself because SEN55 is
optional. The core AIR-1, smart plug, and mmWave fields gate live prediction.

Operator flow:

1. Check `/api/current`.
2. Inspect the `error` field.
3. Verify upstream Smart I-Lab API history for the Zone 5 AIR-1, smart plug,
   and mmWave devices.
4. If upstream data is healthy but the running app cache remains stale, restart
   only the live app service.

Training keeps the latest 1 calendar day as the hidden/blind test. Earlier days
feed adaptive rolling 1-day validation folds, capped at 3 folds. This strict
mode is the normal production standard. If only 2 calendar days are available,
the first day is split chronologically for emergency validation and the second
day stays hidden. Fewer than 2 calendar days will fail because at least 2
calendar days are required for a true 1-day hidden test.

For the first model only, `python -m zone5.training --bootstrap-fallback` can
retry after strict validation finds no viable lookbacks. It keeps the same
hidden test day, uses the first 80% of pre-test rows for training and the last
20% for validation, and still rejects single-class training windows. Promotion
allows that bootstrap candidate only when `model/production_run.txt` does not
already point to a model. Once production exists, future promotion candidates
must come from strict validation. Use `--cv-folds 1`, `--cv-folds 2`, or
`--cv-folds 3` to request the strict rolling fold count explicitly. Live
retraining defaults to `--retrain-cv-folds auto`, which trains 1 fold before
first strict production, then requires 2 folds for the next replacement, then 3
folds after that. One narrow exception lets a strict 1-fold candidate replace
the first strict 1-fold production model when both are `rolling_calendar` runs
and the candidate's stored blind-test PR-AUC is greater than production's
stored blind-test PR-AUC plus `--min-pr-auc-delta`; the next auto retrain still
tries to mature to 2 folds. `--allow-degenerate-validation` is separate and
intended for smoke/dev runs that allow single-class validation windows; it does
not allow single-class training windows, and promotion rejects
degenerate-validation candidates.

The active model contract is `zone5_missingness_decoupled_v1`: raw sensor
columns plus deterministic time features only. Do not deploy old
`zone5_mmwave_recency_v1` artifacts. When retraining for a fixed holdout day,
pass `--blind-test-date YYYY-MM-DD`; rows after that date are excluded from
training/evaluation and recorded in `split_policy`. During an intentional
contract migration, `--force-promote` bypasses loading/comparing the old
production artifact after the new candidate passes its own gates.

## Failure Behavior

The MQTT collectors skip malformed messages and keep running. The SEN55
collector skips payloads for already-flushed 10-second buckets instead of
rewriting historical CSV rows.

If `zone5-person-counter.service` exits with `ModuleNotFoundError: No module
named 'lap'`, reinstall the package requirements into the package-local Python
folder and restart the service. If this keeps happening, the server is still
running an older copy of `run_person_counter.sh`; the updated launcher prints
`Python dependency 'lap' is missing` before opening the camera.

```bash
cd ~/zone5_cv_time_features_package
python3 -m pip install --target .python-packages --upgrade -r requirements.txt
PYTHONPATH="$PWD/.python-packages:$PWD" python3 -c "import lap; print(lap.__file__)"
systemctl --user restart zone5-person-counter.service
systemctl --user status zone5-person-counter.service
```

The person-counter service sets `YOLO_AUTOINSTALL=False`; do not let
Ultralytics auto-install missing packages into `/usr/local` from a user
service. That requires system write permission and will fail with `Permission
denied`.

The live collector retries AIR-1 API timeout and connection failures. If a
historical chunk still fails after retries and adaptive splitting, those sensor
values are blank and the corresponding `*_missing` features are set later.

If CV labels are missing, the joined training table can still be written with
null `zone_occupied` values unless `--require-cv-labels` is used. Training skips
null target windows, but it fails if no labeled target remains.

Promotion refuses artifacts that are not trained against the CV target, fail the
smoke test, or do not beat the existing production model under the configured
thresholds.

The web app stays up when one inference tick fails. It reports the error through
`/api/health`, `/api/current`, and the dashboard.
