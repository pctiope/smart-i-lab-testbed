\column{0.54\textwidth}
\begin{block}{Research setting}
Desk/Zone 5 is a live lab zone, not a fixed benchmark. Occupancy evidence
arrives through CV person counts, AIR-1 environmental readings, smart-plug
power, mmWave state, SEN55 air-quality readings, and time.
\end{block}
\tightsep
\begin{itemize}
  \item Streams can be stale, delayed, partial, or absent while the dashboard remains available.
  \item CV labels are proxy truth and inherit mask, lighting, and occlusion limits.
  \item The model must expose probability and readiness without learning missingness as occupancy.
\end{itemize}
\end{columns}
\end{frame}

\begin{frame}[shrink=4]{Research Questions}
\begin{enumerate}
  \item Can Zone 5 combine environmental, power, mmWave, SEN55, CV, and time evidence without treating missingness itself as occupancy evidence?
  \item Can short-history mmWave recency improve robustness while keeping \code{zone_occupied} CV-derived?
  \item Can BSG DuckDB/Parquet ingestion feed the existing training and promotion contract without changing the serving pointer?
  \item Can the same codebase support production user-systemd services and staged Compose checks without mixing ownership?
\end{enumerate}
\tightsep
\begin{center}
\parbox{0.84\textwidth}{\centering
\small The answer is framed as a complete system walkthrough: data flow, package structure, model contract, training loop, serving endpoints, deployment boundaries, limitations, and source anchors.}
\end{center}
\end{frame}

\begin{itemize}
  \item \code{run_bsg_live_collector.sh} owns live Bronze/Silver ingestion.
  \item Support-table seeding bridges CV and SEN55 rows into BSG.
  \item \code{run_bsg_trainer.sh} snapshots Silver training input.
  \item Promoter owns \code{model/production_run.txt}.
  \item FastAPI owns serving, readiness, and dashboard payloads.
  \item UI/Vite shell is separate from model validity.
\end{itemize}
\end{columns}

\begin{frame}[shrink=4]{Training Table Semantics}
\begin{columns}[T,onlytextwidth]
\column{0.50\textwidth}
\textbf{One row means}
\begin{itemize}
  \item A floored 10-second timestamp for Zone 5.
  \item Latest joined sensor state available for that timestamp.
  \item CV label if the person-count bucket exists.
  \item Missing raw sensor values preserved before preprocessing.
\end{itemize}

\column{0.46\textwidth}
\textbf{One row does not mean}
\begin{itemize}
  \item A camera frame is guaranteed for every sensor row.
  \item Missing CV labels are negative labels.
  \item mmWave is target truth.
  \item A row is promotable evidence before split and coverage checks.
\end{itemize}
\end{columns}
\tightsep
\begin{block}{Review implication}
\small The model claim is a table contract plus validation policy, not just a trained checkpoint.
\end{block}
\end{frame}

\begin{frame}[shrink=4]{Hyperparameter And Training Loop}
\begin{columns}[T,onlytextwidth]
\column{0.50\textwidth}
\textbf{Search space}
\begin{itemize}
  \item Lookback, batch size, learning rate, weight decay.
  \item Number of convolution blocks, base channels, kernel sizes.
  \item Activation, normalization, pooling pattern, global pool.
  \item Dense units, dropout, scheduler, patience.
\end{itemize}

\column{0.46\textwidth}
\textbf{Training mechanics}
\begin{itemize}
  \item Optimizer: AdamW.
  \item Class imbalance: positive weight from train labels.
  \item Schedulers: none, cosine, or reduce-on-plateau.
  \item Objective: mean validation PR-AUC over eligible folds.
  \item Sampler: Optuna TPE with a fixed seed path.
\end{itemize}
\end{columns}

\small This review names the loop and gates but does not embed current run IDs or metric values.

\begin{frame}[shrink=4]{Validation Split Policy}
\begin{columns}[T,onlytextwidth]
\column{0.50\textwidth}
\textbf{Offline split discipline}
\begin{itemize}
  \item Latest calendar day is held out as blind-test evidence.
  \item Earlier eligible dates form one-day rolling validation folds.
  \item Strict mode requires usable train and validation labels.
  \item Bootstrap fallback is first-production recovery, not normal replacement.
\end{itemize}

\column{0.46\textwidth}
\textbf{Lookback viability checks}
\begin{itemize}
  \item A lookback must fit train and validation windows.
  \item Both-class windows are required unless explicitly in smoke/dev mode.
  \item Under-covered dates and rejected lookbacks are recorded in metadata.
\end{itemize}
\end{columns}

\begin{frame}[shrink=4]{Limitations And Threats To Validity}
\begin{columns}[T,onlytextwidth]
\column{0.49\textwidth}
\textbf{Data and label threats}
\begin{itemize}
  \item CV proxy truth can fail under occlusion, lighting shifts, or mask errors.
  \item Sensor/API outages can reduce feature coverage or delay joined rows.
  \item Class imbalance and sparse occupied windows can make metrics unstable.
  \item Delayed buckets can revise recent rows after first append.
\end{itemize}

\column{0.47\textwidth}
\textbf{Operational threats}
\begin{itemize}
  \item Upstream latency can stretch prediction cadence.
  \item Artifact compatibility must be explicit as contracts evolve.
  \item A healthy dashboard is not proof that a candidate is valid.
  \item Scheduled retraining must avoid overlapping live fits.
\end{itemize}
\end{columns}
\end{frame}

\begin{frame}[shrink=4]{Roadmap}
\begin{columns}[T,onlytextwidth]
\column{0.50\textwidth}
\textbf{Near-term hardening}
\begin{itemize}
  \item Continue strict rolling-calendar promotion before expanding scheduled automation.
  \item Track model freshness, source freshness, and label coverage as first-class readiness signals.
  \item Keep mmWave recency monitored for leakage and stale-input behavior.
\end{itemize}

\column{0.46\textwidth}
\textbf{Research extensions}
\begin{itemize}
  \item Quantify feature-family ablations under the missingness-decoupled contract.
  \item Compare recency windows once enough positive evidence accumulates.
  \item Add reviewer-facing diagnostics that summarize split and coverage policy without exposing runtime secrets.
\end{itemize}
\end{columns}
\end{frame}
